# Andrews Trajectory

This ledger establishes Andrews as the active directing and redirecting layer for the engine, visible UI, diagnostics, and curriculum status. It does not create one engine, route, or UI control per lesson. Lessons remain evidence and curriculum indexes over reusable grammar categories.

## Supreme Goal

Follow the trajectory of the Andrews PDF by making Nawat Conjugator a grammar GIS whose authority is the PDF. Lessons are evidence/status waypoints. Andrews governs categories, slots, boundaries, dependencies, source/target routes, generation gates, and whether a route is generated, diagnostic, blocked, or source-gated. Nawat/Pipil may realize spelling through the orthography bridge, but it must not become authority for grammar logic or unsupported derivational routes.

## Patch Judgment Gate

Every future patch must answer three checks before it can be treated as aligned: which Andrews waypoint it advances, which LCM boundary it preserves, and whether Nawat/Pipil authority stays limited to orthographic realization rather than grammar permission. Preterit indicative remains separate from preterit-derived routes such as agentive, adjectival, and embedded preterit behavior.

## Directing Rule

Before a grammar target is implemented, read the relevant Andrews PDF section and let it define the architecture: order, roles, slots, boundaries, categories, dependencies, derivational operations, source/target route, and whether generation is licensed. Nawat/Pipil evidence may record lexical spelling and examples, but it does not decide whether Andrews-licensed grammar logic can generate. Classical spellings remain structural and output must pass through the orthography bridge.

## Redirecting Rule

When current behavior diverges from Andrews, redirect it explicitly. The allowed redirect actions are `keep`, `rename-visible-ui`, `reframe-metadata`, `diagnostic-only`, `block-generation`, `refactor-engine`, and `source-gated`. If a route lacks Andrews license or has an unmet Andrews source gate, prefer diagnostic metadata or blocked generation over silent output.

## Plan/Pursue Rule

Plan Mode aims; Pursue Goal shoots. The main target is the fully Andrews-directed Nawat Conjugador. Each lesson is one step toward that target, and each scoped correction is one arrow inside the current lesson-step.

- Main target: an app whose grammar architecture and route permission are directed by Andrews PDF, with Nawat/Pipil limited to orthographic realization and lexical evidence records.
- Step: one Andrews lesson, ordered from 1 through 58.
- Arrow: one scoped correction to UI, engine, metadata, diagnostics, tests, validation, or docs.
- Hit: the arrow moves the app closer to Andrews and has PDF evidence plus feedback.
- Miss: the arrow lacks Andrews grounding, adds drift, hides uncertainty, or passes irrelevant feedback.
- Closest-pass: the lesson has no known unmarked mismatch; any remaining gap is explicitly blocked, diagnostic-only, or evidence-needed.

Plan Mode must define the step, arrows, hit criteria, miss criteria, and validation before code changes. Pursue Goal must implement only planned arrows, measure each arrow against Andrews plus the orthography boundary, and stop only at closest-pass or an explicit Andrews source-gate blocker.

### Correctness Before Existence Rule

An existing route, formula, label, metadata field, diagnostic, or test is not a hit by existence alone. Before a shot can be kept or marked hit, Andrews' claim must be converted into a concrete behavior requirement and checked through the input-to-output path: entrada/options, parsing, grammar category or route, Andrews-directed slots/order/roles, Nawat orthography or surface handling, diagnostics or generation block, and rendered rows/UI.

Every kept or hit shot must name its miss probe: the wrong output, wrong UI, wrong metadata, or wrong diagnostic that would show the audit only checked existence. A shot is a miss when it exists but violates Andrews order, role, slot ownership, boundary, category, generation permission, orthography scope, Spanish visible UI, or evidence status.

### Andrews Formula Authority Rule

Andrews formulas are engine contracts, not optional explanatory metadata. If Andrews defines a formula category, slot order, slot owner, boundary, or morphic filler, the Conjugador must keep that formula architecture and let Andrews decide whether the route can generate. Nawat/Pipil evidence may confirm lexical spelling records; it does not weaken Andrews formula structure and does not decide Andrews grammar permission.

For CNV Lesson 5, the optative formula is therefore stronger than compatibility behavior: the finite category is `optativo`, the tense/morph slot is Andrews' nonpast optative `Ø`, and every plural optative formula must keep the Nawat-adapted Andrews connector `k-an` in `num1-num2`. A formula that leaves plural optative number as `Ø-t`, `k-et`, a folded subject suffix, or any non-optative connector is a miss even if a surface row renders.

For Lessons 1-4, formula authority is foundational. Lesson 1 treats `#`, parentheses, `+`, and hyphen as notation contracts, not visible Nawat surfaces. Lesson 2 may realize letters only inside the slot and boundary already licensed by Andrews, so the orthography bridge cannot change formula order or slot ownership. Lesson 3 particles remain outside CNV/CNN formula slots unless a later Andrews target explicitly attaches sentence-level particle behavior. Lesson 4 formulas define the active shell contract: CNV may carry `tns`, CNN may not, predicates stay inside parentheses, and `num1-num2` stays a subject-owned connector outside the predicate.

### Formula Surface Coupling Rule

A CNV formula is not correct unless its visible slots match the same slot-level surface rules that produce the output. For every generated CNV output, the validation target is one coupled contract: entrada/options, parsing, grammar route, Andrews slot structure, Nawat/Pipil allomorphy or orthography, surface output, formula echo, diagnostics, and rendered rows.

When a formula is meant to show Nawat-adapted slot realization, render it from the same post-allomorphy slot values and rule predicates that produce the surface segment. A formula is a miss if the output applies a surface rule but the formula still shows the pre-rule slot, if the formula applies a surface rule but the output does not, if either layer uses a broad shortcut instead of the actual surface-rule predicate, if the changed morph is folded into the wrong slot, or if the formula hides a rule that changed the visible output.

Every formula keep or hit must include a paired probe: one case where the relevant rule fires and one case where it does not. Both probes must assert formula slots, surface output, and rule metadata. For example, Lesson 6 reflexive `mu + ajsi` must keep formula `m-u`, output `muajsi`, and no `obj1-mu-before-vowel-m` frame; `mu + altia` must show formula `m-0`, output `maltia`, and the `obj1-mu-before-vowel-m` frame.

### Minimal Andrews-Strong Formula Run Rule

Every lesson run uses the same minimal loop: read the relevant Andrews PDF subsection, split it into each visible subsection and sub-subsection, convert each claim into a behavior requirement, inspect only the entrada-to-output path, and change only a concrete mismatch. Andrews decides formula architecture, route permission, and derivational authority; Nawat/Pipil orthography realizes spelling after Andrews licenses the route.

The completion report for each run must list every inspected subsection and sub-subsection with its Andrews reference, requirement, shot status, miss probe, changed files if any, and validation result. Successful runs allow only `hit-no-edit` or `hit-edit` as final shot statuses. `keep` is a redirect action, not a final shot status; `miss`, `blocked`, or `evidence-needed` mean the lesson has not reached the all-hit target.

## Registry Contract

`src/lessons/registry.js` and `src/lessons/registry.mjs` carry the machine-readable trajectory for each lesson. Each trajectory records:

- `pdfRefs`: direct Andrews lesson or subsection references.
- `directive`: the Andrews grammar rule that directs the implementation.
- `implementationState`: current implementation state.
- `redirectAction`: how current behavior is kept or redirected.
- `evidenceStatus`: whether the claim is audited, partial, diagnostic, or placeholder evidence.
- `orthographyStatus`: whether the route is non-surface-bearing, requires the orthography bridge plus source gates, or has a realized bridge.
- `validationRefs`: focused tests, audits, or validation scripts that keep the claim checked.
- `stepNumber`, `aimStatus`, `plannedArrows`, `firedArrows`, `hitCount`, `missCount`, `remainingGap`, and `closestPass`: the Plan/Pursue state for that lesson-step.
